# API and event contracts

Two contract surfaces: the synchronous HTTP/WebSocket API exposed by `api-service`, and the asynchronous Kafka event bus that connects the internal services. The Kafka envelope is the more important of the two — it is the spine of the system — so it comes first.

---

## Event envelope

Every message on every topic shares one envelope. Payload is topic-specific. Defined once as a Pydantic v2 model in the shared contracts package so producers and consumers cannot drift.

```jsonc
{
  "event_id":   "evt_01HZ...",        // ULID, unique, used for dedupe + provenance
  "org_id":     "org_8f2c",           // tenant; also the Kafka partition key
  "type":       "graph.node.changed", // dotted type, matches topic semantics
  "occurred_at":"2026-07-11T09:14:07Z",// when the real-world thing happened
  "produced_at":"2026-07-11T09:14:07.812Z",
  "producer":   "graph-service@1.4.2",
  "trace_id":   "b7ad...",            // OTel trace id — stitches the whole pipeline
  "schema_version": 1,
  "payload":    { }                    // topic-specific, validated against a registered schema
}
```

Rules: `event_id` is the idempotency key everywhere downstream. `org_id` is the partition key so per-tenant ordering holds and tenants are isolated on the bus. `trace_id` propagates unchanged from the originating source event through to the delivered notification, so one trace shows the full path. `schema_version` allows additive evolution; a consumer rejects a version it does not understand to its DLQ rather than guessing.

## Topic catalog

| Topic | Producer | Consumers | Payload summary |
|---|---|---|---|
| `raw.events` | ingestion-service | entity-service | normalized source event (source, kind, raw ref) |
| `entities.extracted` | entity-service | graph-service | typed entities + candidate relationships |
| `graph.changes` | graph-service | ranking-service, retrieval-service | `{changed_node_ids[], change_kind}` |
| `risk.scored` | ranking-service | llm-service, notification-service | `{node_id, score, features, confidence}` |
| `reasoning.produced` | llm-service | notification-service, api-service | `{explanation, actions[], citations[], confidence}` |
| `notifications.sent` | notification-service | api-service, analytics | `{notification_id, channel, recipients[], fingerprint}` |
| `user.actions` | api-service | graph-service, notification-service | `{action: ack\|dismiss\|snooze, target, actor}` |
| `*.dlq` | any consumer | ops tooling | original message + error + retry count |

Partitioning: all topics keyed by `org_id`. Retention: `raw.events` and `entities.extracted` long enough to replay a full rebuild (e.g. 7d); `graph.changes` short (1d); the event log for the graph itself lives in Neo4j/Postgres, not Kafka, for permanent history.

Delivery: at-least-once with idempotent consumers. Offsets commit only after a successful downstream produce. After N retries a message goes to `<topic>.dlq` with the error attached; it never blocks its partition.

## HTTP API (api-service)

REST under `/api/v1`, OpenAPI/Swagger generated by FastAPI, all routes org-scoped by the JWT. A representative slice, not exhaustive:

```
# Graph
GET  /api/v1/graph/nodes/{id}                 node + immediate edges
GET  /api/v1/graph/nodes/{id}/neighborhood    k-hop subgraph (?hops=2&types=BLOCKS,AFFECTS)
GET  /api/v1/graph/paths?from={id}&to={id}     shortest evidence path(s)
POST /api/v1/graph/query                       constrained Cypher-template query (not raw Cypher)

# Retrieval / search
POST /api/v1/search                            hybrid retrieval {query, anchor?, filters}

# Risk + recommendations
GET  /api/v1/risk/top                          ranked open risks (?limit&min_score)
GET  /api/v1/risk/{node_id}                     score + feature breakdown + why
GET  /api/v1/recommendations                   recommendation feed
GET  /api/v1/timeline?entity={id}              "what changed" temporal view

# Notifications + actions
GET  /api/v1/notifications                     inbox (ranked, bundled)
POST /api/v1/notifications/{id}/ack            → emits user.actions
POST /api/v1/notifications/{id}/dismiss        → emits user.actions
POST /api/v1/notifications/{id}/snooze         → emits user.actions

# Entities
GET  /api/v1/people/{id}   /teams/{id}   /services/{id}   /repositories/{id}
GET  /api/v1/incidents  /projects  /repositories          list + filter

# Admin / connectors
GET  /api/v1/connectors                        status per source (last sync, lag, errors)
POST /api/v1/connectors/{source}/sync          trigger incremental sync
GET  /api/v1/health   /ready   /metrics        liveness, readiness, Prometheus
```

Constrained query note: `/graph/query` accepts parameterized templates, never raw Cypher from the client, to prevent injection and unbounded traversals. Raw exploration is a server-side capability, not a public one.

### Response shape

Uniform envelope so the dashboard handles every endpoint the same way:

```jsonc
{
  "data": { },
  "meta": { "org_id": "org_8f2c", "generated_at": "...", "cached": false },
  "errors": []
}
```

Errors use RFC-9457 problem+json with a stable `type` slug per error class.

## WebSocket / streaming

```
WS  /api/v1/stream                       server-push channel, subscribed by topic
  → subscribe: {"channel": "risk.updates"}      live risk score changes
  → subscribe: {"channel": "notifications"}     new notifications as they route
  → subscribe: {"channel": "graph.updates"}     node/edge deltas for the open explorer view
```

`api-service` bridges selected Kafka topics to WebSocket subscribers, filtered by the connection's `org_id` and the user's role. The dashboard graph explorer uses `graph.updates` to animate changes without polling. LLM explanations can stream token-by-token over the same channel for the "explain this risk" panel.

## Auth

OAuth for connecting sources (GitHub, Google, Slack — PKCE where the provider allows public clients). User sessions are JWTs carrying `org_id`, `sub`, and `roles`. Every request is authorized at the edge and re-scoped at the data layer. Roles: `viewer`, `member`, `admin`, `owner`. Org isolation is structural — see [ADR-0008](../adr/0008-auth-and-tenancy.md).

---

## Contract governance

The envelope, every payload schema, and the shared enums live in one `contracts` package imported by every service. Schemas are versioned; changes are additive within a major version. CI runs a compatibility check that fails a build if a payload change is not backward compatible for existing consumers, so a producer cannot ship a breaking event unnoticed.
